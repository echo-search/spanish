Una actualización sobre nuestro camino de código abierto

Publicado el 24 de diciembre de 2025

La transparencia siempre ha sido un principio fundamental de este proyecto. Desde sus primeros días, tomamos la decisión de publicar el código como código abierto, permitiendo que cualquier persona pudiera inspeccionarlo, aprender de él y contribuir a su desarrollo. Esa elección moldeó no solo cómo evolucionó el proyecto técnicamente, sino también cómo fue percibido y utilizado.

Hoy queremos comunicar claramente un cambio importante. Después de un año completo siendo de código abierto, el proyecto pasará a un modelo cerrado.

A partir del 3 de junio de 2026, este proyecto dejará de ser de código abierto.

Este artículo explica por qué ocurre este cambio, qué significa para los usuarios y colaboradores, y cómo estamos abordando la transición.

El propósito de ser de código abierto

Cuando el proyecto se lanzó por primera vez, todavía era experimental y cambiaba rápidamente. Abrir el código permitió recibir retroalimentación rápida, revisión externa y mejoras orgánicas. Los desarrolladores podían explorar cómo funcionaban las funciones internamente, sugerir optimizaciones e identificar errores temprano.

El código abierto también tuvo un papel educativo. Al hacer el código accesible públicamente, permitimos que otros estudiaran implementaciones reales, decisiones de arquitectura y compromisos técnicos. Esa apertura ayudó a construir confianza y credibilidad en las primeras etapas del proyecto.

Durante el año, el proyecto se benefició de reportes de errores, discusiones, forks y experimentación de la comunidad. Incluso cuando los cambios no se incorporaban directamente, los aprendizajes obtenidos fueron valiosos.

Cómo ha evolucionado el proyecto

Durante el último año, el alcance y la madurez del proyecto han aumentado significativamente. Lo que comenzó como una base de código relativamente pequeña se ha convertido en un sistema más complejo con más dependencias, integraciones más profundas y mayores expectativas de rendimiento y fiabilidad.

Con el aumento del uso también aumentaron las necesidades de seguridad, consistencia y mantenimiento a largo plazo. Estas necesidades requieren un enfoque de desarrollo diferente al que funciona en etapas tempranas de experimentación.

En esta etapa, el acceso público sin restricciones al código fuente completo ya no es la forma más efectiva de apoyar la dirección del proyecto o a sus usuarios.

Por qué pasamos a un modelo cerrado

La decisión de cerrar el código no se tomó a la ligera. El código abierto tiene muchas ventajas, pero también introduce desafíos que se hacen más evidentes a medida que el proyecto crece.

Mantener un repositorio abierto requiere más trabajo: gestionar pull requests externos, manejar forks que pueden representar mal el proyecto, responder a reportes de seguridad en público y evitar que trabajo interno o incompleto sea malinterpretado o mal utilizado.

Pasar a un modelo cerrado permite centrarnos más en la estabilidad, el rendimiento y un diseño coherente. También ayuda a proteger detalles de implementación y reduce el riesgo de fragmentación o uso indebido.

Qué significa este cambio en la práctica

Hasta el 3 de junio de 2026, el proyecto seguirá siendo de código abierto bajo sus condiciones actuales. Durante este tiempo, el repositorio seguirá accesible y las contribuciones existentes seguirán siendo reconocidas.

Después de esa fecha, el repositorio público dejará de actualizarse y el código fuente dejará de estar disponible públicamente. El desarrollo futuro continuará de forma privada. El repositorio de GitHub pasará a ser privado. Sin embargo, el repositorio de planificación, Add to EchoSearch, seguirá siendo público y siempre lo será.

Esta transición no significa el fin del producto o del servicio. Los usuarios podrán seguir usando la plataforma normalmente. El cambio solo afecta a la forma en que se distribuye el código fuente.

Reconociendo el papel de la comunidad

La participación en código abierto tuvo un papel importante en la evolución del proyecto durante su primer año. Cada reporte, discusión y sugerencia contribuyó a su mejora, de forma directa o indirecta.

Reconocemos que algunos usuarios valoran mucho el software de código abierto y pueden sentirse decepcionados con este cambio. Esa perspectiva es válida, y queremos ser claros y honestos en lugar de ambiguos.

La decisión de pasar a un modelo cerrado no es una falta de respeto hacia la apertura, sino una forma de garantizar sostenibilidad y calidad a largo plazo.

Mirando hacia adelante

A medida que el proyecto avanza, nuestras prioridades son claridad, fiabilidad y crecimiento responsable. Un modelo cerrado nos da flexibilidad para iterar más rápido internamente mientras mantenemos una experiencia estable para los usuarios.

Seguimos comprometidos con la transparencia donde más importa: comunicar cambios, explicar decisiones y establecer expectativas claras. Este artículo forma parte de ese compromiso.

El capítulo de código abierto de este proyecto fue importante. Cumplió su propósito, ayudó a madurar el proyecto y sentó las bases de lo que viene.
